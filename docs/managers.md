# Configuring the manager

The manager keeps one or more dashboard YAML config files in sync. Pick which dashboards it
manages, where their config files live on disk, and any per-service defaults you want, all in
[`manager-config.yaml`](../docker-compose/manager-config.yaml).

## What the manager writes

For every service the provider posts, the manager:

1. Filters out anything with `Ignore: true`, an empty URL, or a name in `IgnoredServiceNames`.
2. Applies per-service defaults — adding an icon, category, or hostname-templated name — from
   the shipped defaults plus your overrides.
3. For each configured dashboard, reads the existing file, merges in the new entries
   (preserving anything you typed by hand), and writes the result back. Identical content is
   **not** rewritten, so dashboards that watch the file don't reload spuriously.

The merge rule is the load-bearing one: anything the manager didn't put there itself stays put.
You can hand-edit your dashboard YAML and the manager won't touch your changes — it only
replaces entries it owns.

## Dashboard backends

### Dashy

```yaml
- DashboardManagerType: Dashy
  DashboardConfigDirectoryPath: /dashy_config       # writes /dashy_config/conf.yml
```

The manager rewrites only the `sections:` block; everything else in the file (`appConfig`,
`pageInfo`, theme overrides, etc.) is preserved. Auto-generated items get an `_autogenerated`
tag plus a `host=<hostname>` tag, so you can spot them in the UI:

```yaml
items:
  - title: Jellyfin
    url: https://jellyfin.example
    tags:
      - _autogenerated
      - host=media-01
```

If you remove the `_autogenerated` tag from an entry, the manager treats it as user-owned from
that point on — re-runs of the provider won't replace it.

### Homepage

```yaml
- DashboardManagerType: Homepage
  DashboardConfigDirectoryPath: /homepage_config    # writes /homepage_config/services.yaml
```

The manager rewrites `<dir>/services.yaml` and updates `<dir>/settings.yaml` with category
icons (non-icon settings — theme, providers, layout columns, etc. — are preserved).
Auto-generated entries are marked with a custom `_autogenerated: true` field that Homepage
ignores but the manager uses on the next run:

```yaml
- Media:
    - Jellyfin:
        href: https://jellyfin.example.com
        icon: jellyfin.png
        server: media-01
        _autogenerated: true
```

Removing `_autogenerated: true` from an entry does the same thing as removing the tag in
Dashy: the entry becomes user-owned and is preserved going forward.

## Status monitoring

Per-dashboard flag — when on, every entry gets the dashboard's status-ping directive:

```yaml
- DashboardManagerType: Homepage
  DashboardConfigDirectoryPath: /homepage_config
  EnableStatusMonitoring: true
```

| Dashboard | What gets emitted |
|---|---|
| Dashy | `statusCheck: true` per item ([Dashy status indicators](https://dashy.to/docs/status-indicators)) |
| Homepage | `siteMonitor: <href>` per entry ([Homepage site monitor](https://gethomepage.dev/configs/services/#site-monitor)) |

For Homepage specifically: if a service has a widget attached (see below), `siteMonitor` is
suppressed for that entry — widgets do their own polling and emitting both would be redundant.

## Homepage service widgets

Optional. Point `SourceHomepageServiceWidgetsFilePath` at a YAML file in
[Homepage's service-widgets format](https://gethomepage.dev/configs/service-widgets/) and the
manager will merge the matching widget into each service entry it writes.

```yaml
- DashboardManagerType: Homepage
  DashboardConfigDirectoryPath: /homepage_config
  SourceHomepageServiceWidgetsFilePath: /homepage_config/widgets.yaml
```

A widgets file with three AdGuard Home instances on different hosts, plus an unscoped Immich
widget:

```yaml
- adguardhome:
    server: network-01
    widget:
      type: adguard
      url: https://adguardhome.example
      username: "{{ network_01_adguardhome_username }}"
      password: "{{ network_01_adguardhome_password }}"
- adguardhome:
    server: pi-rack
    widget:
      type: adguard
      url: https://adguardhome-02.example
- adguardhome:
    server: network-03
    widget:
      type: adguard
      url: https://adguardhome-03.example
- immich:
    widget:
      type: immich
      url: https://immich.example
      key: ...
```

Match precedence (case-insensitive on both sides):

1. **Service name + hostname** — top-level key matches the service name and `server:` matches
   the host the service runs on.
2. **Default service name + hostname** — for services renamed by a hostname-templated name
   (e.g. `adguardhome` → `AdGuard Home (network-01)`), the widget can still key off the
   original short name. Combined with `server:`, this lets one widgets file map cleanly to
   multiple hosts running the same service.
3. **Service name only** — widget without `server:`, used when the hostname doesn't match any
   server-scoped entry.
4. **Default service name only** — final fallback.

Widgets without a `server:` work as a catch-all for any host that doesn't have a more specific
match.

## Per-service defaults

DSM ships with a default set of icon prefixes, category mappings, and name templates for popular
services (Plex, AdGuard Home, Traefik, Proxmox, etc.). Override or extend them in
`manager-config.yaml` under `ServiceDefaultOptions`:

```yaml
ServiceDefaultOptions:
  Categories:
    media:    { Icon: fas fa-tv }
    network:  { Icon: fas fa-network-wired }
  Services:
    plex:
      Icon: my-custom-plex-icon         # overrides shipped icon
    my-internal-tool:
      Category: utilities
      ImagePath: /assets/logo.svg       # resolved against the service URL
```

Per-field rule: anything the provider sent wins; defaults fill in the gaps. Removing a default
means setting the corresponding key blank — the merge is leaf-level, so unrelated keys inside
the same service entry survive your override.

Icons can also come from a CDN. Prefix the icon name to pick a source:

- `hl-` — [Homarr Labs dashboard-icons](https://github.com/homarr-labs/dashboard-icons), e.g.
  `Icon: hl-jellyfin`.
- `sh-` — [selfh.st icons](https://github.com/selfhst/icons), e.g. `Icon: sh-sonarr`.
- `mdi-` — [Material Design Icons](https://pictogrammers.com/library/mdi/) via the `@mdi/svg`
  package, e.g. `Icon: mdi-account`.

The manager probes the CDN and writes a URL into the dashboard entry's image field, clearing
the literal icon name. If you want the literal `mdi-X` written through to your dashboard YAML
(so Dashy or Homepage renders MDI natively, instead of fetching the SVG from the CDN), don't
use the `mdi-` prefix — set the icon a different way, e.g. via a category icon under
`Categories.<n>.Icon` (those bypass the prefix resolver).

If a service ends up without an icon and without an image URL, the manager probes the
`FallbackIconSourceProviders` chain (default: Homarr Labs, then selfh.st) using the service's
name. Set this list to `[]` to disable the fallback.

## Ignored services

Some service names should never appear on a dashboard — typically the manager itself, or other
infrastructure services that the provider scrapes from a container label even though you don't
want them surfaced.

```yaml
ManagerOptions:
  IgnoredServiceNames:
    - Dashboard Services Manager
    - Dashy
    - Homepage
```

Comparison is case-insensitive against the service name.

## Multiple dashboards

You can configure several `DashboardManagers` entries — one per dashboard target. The manager
writes the same merged service list into each, independently:

```yaml
ManagerOptions:
  DashboardManagers:
    - DashboardManagerType: Dashy
      DashboardConfigDirectoryPath: /dashy_config
    - DashboardManagerType: Homepage
      DashboardConfigDirectoryPath: /homepage_config
      EnableStatusMonitoring: true
```

Two of the same `DashboardManagerType` is **not** supported — the manager fails fast at
startup if it sees a duplicate.
