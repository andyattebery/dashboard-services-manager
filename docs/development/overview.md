# Architecture overview

Dashboard Services Manager (DSM) keeps a self-hosted dashboard's YAML config file in sync with
the services actually running in your infrastructure. Two long-lived processes work together:

- **[Dsm.Manager.Api](../../src/Dsm.Manager.Api/)** ‚Äî ASP.NET Core Web API. Owns the dashboard YAML
  file (read/write), applies per-service defaults, and runs the merge that preserves hand-edited
  entries.
- **[Dsm.Provider.App](../../src/Dsm.Provider.App/)** ‚Äî console `BackgroundService`. Polls one or
  more providers (Docker, Docker Swarm, Traefik, YAML file) and POSTs the discovered services to
  the Manager API.

Everything else is either a library consumed by those two (`Dsm.Managers`, `Dsm.Providers`,
`Dsm.Shared`) or a test project.

## What problem this solves

Dashy and Homepage configs are usually hand-maintained YAML ‚Äî fine when you have ten services and
they never change, painful past that. Containers come and go, services move between hosts, and
the dashboard drifts out of sync. The naive fix is to regenerate the file from scratch on every
push, but dashboards aren't *just* a list of containers: they hold external links, manual
descriptions, custom group ordering, items that have no infrastructure backing at all.
Regenerating the file loses all of that.

DSM splits the dashboard into two populations of entries: ones DSM put there (stamped with
`_autogenerated: true` on Homepage, or an `_autogenerated` tag on Dashy) and ones a human typed.
On every push it reads the existing file, drops *only* its own previous entries when an incoming
service overlaps with them by name or hostname, and writes the union back. Hand-edited entries
are load-bearing ‚Äî the combiner is the load-bearing piece of the system, and the rest of
[managers.md](managers.md) hangs off that rule.

A second design choice: the manager is the only writer. Providers don't touch dashboard files
directly ‚Äî they just describe what they see and POST it. That way you can run as many providers
as you have hosts (one per Docker daemon, plus one for a static YAML file, plus one talking to
Traefik) and they all fan into the same merged YAML.

## Data flow

```
[infra: Docker / Swarm / Traefik / YAML file]
        ‚îÇ  IServicesProvider.ListServices()
        ‚ñº
Dsm.Provider.App  (BackgroundService, ticks every RefreshInterval)
        ‚îÇ  POST /dashboard-services  (List<Service>, JSON via Refit)
        ‚ñº
Dsm.Manager.Api ‚Üí DashboardCommandProcessor
        ‚îÇ  filter (Ignore / blank URL / IgnoredServiceNames)
        ‚îÇ  mark Autogenerated = true
        ‚îÇ  apply defaults (ServiceWithDefaultsFactory)
        ‚îÇ  for each configured manager:
        ‚îÇ     read existing ‚Üí combine ‚Üí write (skipped when bytes match)
        ‚ñº
IDashboardManager (Dashy, Homepage) ‚Üí <dashboard-dir>/conf.yml or <dashboard-dir>/services.yaml
```

One POST in detail:

1. **Filter** ‚Äî drop incoming services with `Ignore = true`, blank `Url`, or names in
   `ManagerOptions.IgnoredServiceNames` (case-insensitive).
2. **Stamp `Autogenerated = true`** ‚Äî the marker the combiner uses to tell pushed entries from
   hand-written ones.
3. **Apply defaults** ‚Äî every survivor goes through
   [`ServiceWithDefaultsFactory`](../../src/Dsm.Managers/Services/ServiceWithDefaultsFactory.cs):
   per-field precedence (incoming wins, defaults fill gaps), name templating, `ImagePath`
   resolution. Icon-prefix lookup is **not** done here anymore ‚Äî it moved to a per-manager
   step (see #4 below). Full rules in [service-defaults.md](service-defaults.md).
4. **Per-manager fan-out** ‚Äî for each entry in `ManagerOptions.DashboardManagers`:
   - **Read** the existing dashboard file.
   - **Merge** old + new with [`ServicesCombiner`](../../src/Dsm.Managers/Services/ServicesCombiner.cs).
     User-added items always survive; autogenerated items get replaced when an incoming service
     claims the same name or hostname.
   - **Resolve icons** for each merged service against this manager's
     [`NativeIconSourcePrefixes`](../../src/Dsm.Managers/DashboardManagers/IDashboardManager.cs)
     via [`IconResolver`](../../src/Dsm.Managers/Services/IconResolver.cs). Native prefixes
     pass through; non-native ones probe the CDN.
   - **Write** ‚Äî but only if the serialized YAML differs from what's on disk. No-op writes are
     suppressed so file watchers (Dashy, Homepage) don't reload spuriously.

Wire-shape note: because icon-prefix resolution moved out of the factory, the `Service` shape
the system carries (and the one returned from `GET /dashboard-services`) keeps prefix-form
icons (`mdi-account`, `hl-jellyfin`) instead of pre-resolved CDN URLs in `ImageUrl`. The
prefix-to-URL retranslation now happens only at the moment of writing each dashboard's YAML.

The HTTP response is a per-manager `Dictionary<string, List<Service>>` keyed by manager type
(`"Dashy"`, `"Homepage"`) ‚Äî the merged list each manager just wrote, so per-manager visibility is
meaningful for debugging.

`GET /dashboard-services` is a different shape: a single flat deduped `List<Service>` across all
configured managers, deduped by `(Name, Hostname)`. See [managers.md](managers.md) for why the
read shape diverges from the write shape.

The wire contract is [`Service`](../../src/Dsm.Shared/Models/Service.cs); Refit serializes it as
JSON.

## Configuration at a glance

DSM reads three layers, later wins:

1. **Shipped defaults** ‚Äî [service-defaults.yaml](../../src/service-defaults.yaml) (Manager API only,
   required). Defines the built-in `ServiceConfig` overlays (icons, categories,
   `NameWithHostnameFormatString`) and the `FallbackIconSourceProviders` chain. See
   [service-defaults.md](service-defaults.md).
2. **User config** ‚Äî [manager-config.yaml](../../docker-compose/manager-config.yaml) for the API
   and [provider-config.yaml](../../docker-compose/provider-config.yaml) for the worker. Both
   optional. The API loads
   `manager-config.yaml` from the binary directory and from `/config/`; the worker does the same
   with `provider-config.{yml,yaml}`. Bind into `ManagerOptions` / `ServiceDefaultOptions` /
   `ProviderOptions` respectively.
3. **Environment variables** ‚Äî `DSM_*` prefix, double-underscore for nesting
   (e.g. `DSM_ProviderOptions__Hostname=media-01`). Override anything from the YAML layer.

YAML files load through
[`AddNormalizedYamlFile`](../../src/Dsm.Shared/Configuration/NormalizedYamlConfigurationSource.cs),
which strips underscores from keys at load time ‚Äî `snake_case`, `camelCase`, and `PascalCase`
all bind. Dashes in dict keys (e.g. `calibre-web`) are preserved.

The HTTP surface is small:
[DashboardController](../../src/Dsm.Manager.Api/Controllers/DashboardController.cs) exposes
`POST /dashboard-services` (update) and `GET /dashboard-services` (list).

## Build / Run / Test

The production path is Docker Compose. [docker-compose/docker-compose.yaml](../../docker-compose/docker-compose.yaml)
references the published images at `ghcr.io/andyattebery/dashboard-services-manager:{api,provider}-latest`
(built and pushed by [.github/workflows/ci.yaml](../../.github/workflows/ci.yaml) on every push to
`main`). Copy that file plus `env`, `manager-config.yaml`, and `provider-config.yaml` to a deploy
host, then:

```sh
docker compose --env-file env pull
docker compose --env-file env up -d
```

For local development from a working tree, [docker-compose/docker-compose.override.yaml](../../docker-compose/docker-compose.override.yaml)
adds `build:` directives pointing at [docker/api.Dockerfile](../../docker/api.Dockerfile) and
[docker/provider.Dockerfile](../../docker/provider.Dockerfile). Compose auto-merges it when both
files are in the same directory, so devs run:

```sh
cd docker-compose
docker compose --env-file env up --build -d
```

The override file is dev-only ‚Äî production hosts that copied just the four production files
never see it.

For the .NET dev loop without containers, run the projects directly from the repo root:

```sh
# build
dotnet build src/Dsm.sln

# run the API (defaults: http://localhost:5270, https://localhost:7015)
dotnet run --project src/Dsm.Manager.Api

# run the provider worker against a running API
DSM_ProviderOptions__ApiUrl=http://localhost:5270 \
DSM_ProviderOptions__ServicesProviders__0__ServicesProviderType=Docker \
DSM_ProviderOptions__ServicesProviders__0__Hostname=media-01 \
DSM_ProviderOptions__ServicesProviders__0__DockerLabelPrefix=dsm \
DSM_ProviderOptions__ServicesProviders__0__AreServiceHostsHttps=true \
dotnet run --project src/Dsm.Provider.App

# tests (skip jsDelivr-icon tests that hit the public CDN)
dotnet test src/Dsm.sln --filter "TestCategory!=Network"
```
