# Dsm.Managers

Owns the dashboard file. Everything that reads or writes the dashboard config, applies per-service
defaults, or decides what happens when the provider's view of the world disagrees with what a human
has typed into the dashboard lives here.

See [overview.md](overview.md) for how this fits into the larger system.

## Entry points (CQRS split)

Two narrow public services are the only things callers outside this project should reach for. Both
are resolved from DI by [DashboardController](../Dsm.Manager.Api/Controllers/DashboardController.cs).

- **[DashboardCommandProcessor](../Dsm.Managers/DashboardCommandProcessor.cs)** ‚Äî writes.
  `UpdateWithServicesFromProvider(IEnumerable<Service>)` is the one method that drives the whole
  merge/write pipeline. Returns `Dictionary<string, List<Service>>` keyed by manager type
  (`"Dashy"`, `"Homepage"`) ‚Äî each manager gets its own hand-written entries back, so per-manager
  visibility is meaningful on the write path.
- **[DashboardQueryService](../Dsm.Managers/DashboardQueryService.cs)** ‚Äî reads.
  `ListServices()` returns a single flat `List<Service>`, deduped across configured managers by
  `(Name, Hostname)`. Every manager *should* hold the same set of services (the processor just fanned
  the same combined list out to each), so surfacing a per-manager dict here would expose drift
  that's rarely meaningful. First manager in `DashboardManagers` wins on conflict.

Both delegate to an [`IDashboardManager`](../Dsm.Managers/DashboardManagers/IDashboardManager.cs)
picked by [`DashboardManagerFactory`](../Dsm.Managers/DashboardManagers/DashboardManagerFactory.cs).
Two implementations ship today:
[`DashyDashboardManager`](../Dsm.Managers/DashboardManagers/DashyDashboardManager.cs) and
[`HomepageDashboardManager`](../Dsm.Managers/DashboardManagers/HomepageDashboardManager.cs) ‚Äî see
[Implementations](#implementations) below.

```csharp
public interface IDashboardManager
{
    Task<List<Service>> ListServices();
    Task WriteServices(List<Service> services);
}
```

## Request flow (write path)

`POST /dashboard-services` ‚Üí
[DashboardController.UpdateWithServices](../Dsm.Manager.Api/Controllers/DashboardController.cs#L22) ‚Üí
[DashboardCommandProcessor.UpdateWithServicesFromProvider](../Dsm.Managers/DashboardCommandProcessor.cs#L28):

1. **Filter** incoming services ‚Äî drop anything with `Ignore = true`, blank URL, or a name in
   `ManagerOptions.IgnoredServiceNames` (case-insensitive).
2. **Mark autogenerated** ‚Äî every surviving service gets `Autogenerated = true` so the combiner can
   distinguish "the tool pushed this" from "a human typed this."
3. **Apply defaults** via
   [WithDefaultsServiceFactory.CreateWithDefaults](../Dsm.Managers/Factories/WithDefaultsServiceFactory.cs#L19):
   name templating (`NameWithHostnameFormatString`), category, icon, `ImagePath` (absolute
   passthrough or URI-join against `service.Url`), and a walkxcode CDN icon fallback.
4. **For each configured manager** (`ManagerOptions.DashboardManagers`), fan out:
   a. **Read** the current dashboard via `IDashboardManager.ListServices()`.
   b. **Merge** old + new with
      [ServicesCombiner.CombineExistingServicesWithNewServices](../Dsm.Managers/Factories/ServicesCombiner.cs#L7).
   c. **Write** the merged list via `IDashboardManager.WriteServices(...)`.

Steps 1‚Äì3 run once. Step 4 runs per-manager ‚Äî each dashboard has its own file and its own
hand-written entries, so the combiner runs against each one independently. The response is a
per-manager `Dictionary<string, List<Service>>` keyed by manager type name (`"Dashy"`,
`"Homepage"`). If the same `DashboardManagerType` shows up more than once in `DashboardManagers`,
the loop throws `InvalidOperationException` ‚Äî per-entry disambiguation (a `Name` field) is a
future change.

`GET /dashboard-services` is not the same shape: it returns a flat deduped `List<Service>`. See
the `DashboardQueryService` bullet above.

## The combiner rule (load-bearing)

User-added items **always** survive. Autogenerated items are dropped only when a new item claims
the same title or hostname. This is the whole reason the tool doesn't clobber hand-edited dashboards:

```text
newTitles    = {s.Name      for s in newServices}
newHostnames = {s.Hostname  for s in newServices if s.Hostname is not blank}

result = [...newServices]
for existing in existingServices:
    if not existing.Autogenerated
       or (existing.Name    not in newTitles
           and existing.Hostname not in newHostnames):
        result.append(existing)

return result sorted by (Category, Name)
```

The sort at the end keeps the dashboard YAML stable across writes ‚Äî no spurious diffs just from
dictionary ordering.

## Implementations

Each `IDashboardManager` translates between DSM's internal `Service` shape and a specific
dashboard's config format. Both current implementations expose DSM-ownership as an
`_autogenerated` marker so the combiner rule above can distinguish pushed entries from
hand-written ones ‚Äî but they carry that marker differently.

### Dashy

Dashy stores a typed `sections: [...]` block. DSM rewrites just that block; every other top-level
key is preserved.

The `_autogenerated` marker round-trips as an item-level tag:

```yaml
items:
  - title: Jellyfin
    url: https://jellyfin.example
    tags:
      - _autogenerated
      - host=media-01
```

On read,
[DashyDashboardManager.ListServices](../Dsm.Managers/DashboardManagers/DashyDashboardManager.cs#L26)
sets `Service.Autogenerated = tags.Contains("_autogenerated")`. On write,
[DashyItem.Create](../Dsm.Managers/DashboardManagers/Dashy/DashyItem.cs#L9) always stamps
`_autogenerated` plus a `host=...` tag. A user-added item that lacks the `_autogenerated` tag is,
by definition, preserved by the combiner.

### Homepage

[Homepage](https://gethomepage.dev/configs/services/) uses a separate `services.yaml` file
(alongside `settings.yaml`, `bookmarks.yaml`, `docker.yaml`, etc.). Its shape is a list of
single-key group maps, each holding a list of single-key service maps:

```yaml
- Media:
    - Jellyfin:
        href: https://jellyfin.example.com
        icon: jellyfin.png
        server: media-01-docker
        _autogenerated: true
- Links:
    - Personal Blog:
        href: https://blog.example.com
        icon: mdi-rss
```

[`HomepageDashboardManager`](../Dsm.Managers/DashboardManagers/HomepageDashboardManager.cs)
points at the `services.yaml` via the `DashboardConfigFilePath` on its `DashboardManagers` entry.
Field mapping:

| `Service` | Homepage field |
|---|---|
| `Name` | entry key (`- Jellyfin:`) |
| `Category` | group key (`- Media:`), fallback `Uncategorized` on write |
| `Url` | `href` |
| `Icon` / `ImageUrl` | `icon` ‚Äî split on read: `^http‚Ä¶` lands in `ImageUrl`, anything else in `Icon`. Write picks `ImageUrl ?? Icon`, omits the field when both are null |
| `Hostname` | `server` ‚Äî a reference to a key in the user's `docker.yaml` |
| `Autogenerated` | custom `_autogenerated: true` field (unknown to Homepage but ignored) |

DSM never writes a `container:` field today (`Service` doesn't carry a container name).
`WriteServices` rewrites the whole file: entries with `_autogenerated: true` are dropped and
replaced with the incoming services; any other entry is preserved in place, so hand-edited
Homepage services stay untouched.

## Defaults from `manager-config.yaml`

Bound by [HostBuilderConfiguration](../Dsm.Managers/Hosting/HostBuilderConfiguration.cs) into
[`ManagerOptions`](../Dsm.Managers/Configuration/ManagerOptions.cs) and
[`ServiceDefaultOptions`](../Dsm.Managers/Configuration/ServiceDefaultOptions.cs). The shape is:

```yaml
ManagerOptions:
  DashboardManagers:
    - DashboardManagerType: dashy
      DashboardConfigFilePath: dashy_conf.yml
    - DashboardManagerType: homepage
      DashboardConfigFilePath: services.yaml
  IgnoredServiceNames: []

ServiceDefaultOptions:
  UseWalkxcodeDashboardIcons: true
  Categories:
    media:   { icon: fas fa-tv }
    network: { icon: fas fa-network-wired }
  Services:
    traefik:
      NameWithHostnameFormatString: "Traefik [%s]"   # "%s" ‚Üí service.Hostname
      category: network
    grafana:
      category: monitoring
      ImagePath: /public/img/grafana_icon.svg        # joined against service.Url
```

Keys are PascalCase on the outer objects (`ManagerOptions`, `ServiceDefaultOptions`,
`NameWithHostnameFormatString`, `ImagePath`) because `Microsoft.Extensions.Configuration` does a
case-insensitive match but does **not** convert snake_case to PascalCase. Lowercase keys inside the
per-service dictionaries (`category`, `icon`) still bind because the match is case-insensitive.

## Adding a new dashboard type

Four changes, no surprises:

1. Implement [`IDashboardManager`](../Dsm.Managers/DashboardManagers/IDashboardManager.cs). Take a
   [`DashboardManagerConfig`](../Dsm.Managers/Configuration/DashboardManagerConfig.cs) as a ctor
   parameter and read the target file from `config.DashboardConfigFilePath` ‚Äî the factory injects
   that value via `ActivatorUtilities.CreateInstance`, so the manager never needs `IOptions<ManagerOptions>`.
2. Add a variant to
   [`DashboardManagerType`](../Dsm.Managers/DashboardManagers/DashboardManagerType.cs).
3. Extend the `switch` in
   [`DashboardManagerFactory.Create(DashboardManagerConfig)`](../Dsm.Managers/DashboardManagers/DashboardManagerFactory.cs)
   to hand the config to your new manager via `ActivatorUtilities.CreateInstance<T>`.

Expose autogenerated state in whatever form your target config can carry ‚Äî Dashy uses an item
tag, Homepage uses a custom field ‚Äî then set `Service.Autogenerated` on read and stamp the
marker on write. That's the contract the combiner rule leans on.
